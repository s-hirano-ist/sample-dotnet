# CIを理解する

CIはContinuous Integration（継続的インテグレーション）の略です。

コードをリポジトリへpushしたときやPull Requestを作成したときに、自動でテストやビルドを実行します。

## 1. なぜCIが必要か

ローカル環境で動いていても、他の環境で次の問題が起きる可能性があります。

- テストを実行し忘れている
- Dockerfileが壊れている
- Composeの設定が不正
- 他の変更で既存機能が壊れている

CIは、変更を共有する前に機械的な検証を行います。

```text
コードをpush
  ↓
GitHub Actions
  ↓
restore、test、Compose検証、Docker build
  ↓
成功または失敗を表示
```

## 2. Workflow

GitHub Actionsの設定は`.github/workflows/ci.yml`に置きます。

```yaml
name: CI

on:
  push:
  pull_request:
```

`on`は、Workflowを開始するイベントです。

このプロジェクトでは次のタイミングで実行します。

- `main`へのpush
- Pull Request
- GitHub画面からの手動実行

## 3. Job

Workflowは複数のJobに分けられます。

```text
test
compose-config
container
```

Jobは通常、別々の実行環境で動きます。

## 4. .NETテストJob

`test` Jobでは、SDKを準備してテストを実行します。

```yaml
- name: Setup .NET
  uses: actions/setup-dotnet@v4
  with:
    dotnet-version: 10.0.x

- name: Restore
  run: dotnet restore

- name: Test
  run: dotnet test --configuration Release --no-restore
```

ローカルで実行していたコマンドを、GitHub上のクリーンな環境でも実行します。

## 5. Compose設定の検証

```yaml
- name: Validate compose.yaml
  run: docker compose config --quiet
```

`docker compose config`は、Composeファイルの構文と設定を解決します。コンテナを起動しなくても、YAMLの構成ミスを検出できます。

## 6. Dockerイメージのビルド

```yaml
- name: Build image
  uses: docker/build-push-action@v6
  with:
    context: .
    push: false
    tags: todo-api:ci
```

このJobではイメージをビルドしますが、レジストリへのpushはしません。

```text
push: false
```

本番デプロイまで進める場合は、別のJobでレジストリへpushし、ECSなどへデプロイします。

## 7. Job間の依存

コンテナビルドJobには、次の指定があります。

```yaml
needs: test
```

これは、テストJobが成功してからコンテナビルドを実行する指定です。

```text
test成功
  ↓
container build
```

テストが失敗した場合、コンテナイメージはビルドされません。

## 練習問題

次の変更があったとき、どのJobで検出できるか考えてみてください。

```text
TodoServiceのコードがコンパイルできない
compose.yamlのインデントが壊れている
Dockerfileのpublish処理が失敗する
```

確認するポイント:

- `test` Jobの役割
- `compose-config` Jobの役割
- `container` Jobの役割
- `needs`が保証する実行順
