# レート制限を理解する

レート制限は、一定時間内に受け付けるリクエスト数を制限する仕組みです。

## 1. なぜ必要か

レート制限がないと、1つのクライアントが短時間に大量のリクエストを送れます。

```text
クライアント
  ↓ 大量リクエスト
API
  ↓
DBやCPUへ負荷が集中
```

レート制限は、過剰アクセスや意図しない大量リクエストからAPIを守るために使います。

## 2. 固定ウィンドウ方式

現在のMemoryモードでは、固定ウィンドウ方式を使っています。

```json
{
  "RateLimit": {
    "PermitLimit": 10,
    "WindowSeconds": 10
  }
}
```

これは次の意味です。

```text
10秒間に10リクエストまで許可
11回目以降は拒否
ウィンドウが終わるとカウンターをリセット
```

## 3. パーティション

すべてのクライアントを1つのカウンターで数えるのではなく、クライアントごとに状態を分けます。

```text
クライアントA -> カウンターA
クライアントB -> カウンターB
クライアントC -> カウンターC
```

このAPIでは、認証済みならユーザー名、未認証ならIPアドレスをキーにします。

```csharp
var partitionKey = httpContext.User.Identity?.IsAuthenticated == true
    ? $"user:{httpContext.User.Identity.Name}"
    : $"ip:{httpContext.Connection.RemoteIpAddress}";
```

## 4. 429 Too Many Requests

制限を超えたリクエストには、`429`を返します。

```http
HTTP/1.1 429 Too Many Requests
```

これはサーバーが壊れたという意味ではなく、「今はリクエストを受け付けられない」という意味です。

## 5. Retry-After

`Retry-After`ヘッダーで、クライアントへ再試行の目安を伝えます。

```http
Retry-After: 10
```

クライアントは、すぐに繰り返すのではなく、指定された時間を待つ設計にできます。

## 6. インメモリの制限

Memoryモードでは、制限のカウンターはアプリケーションプロセス内にあります。

```text
コンテナA -> カウンターA
コンテナB -> カウンターB
```

複数コンテナで動かすと、カウンターは共有されません。各コンテナが個別に制限します。

## 7. Redisによる分散制限

Redisモードでは、複数コンテナから共通のRedisカウンターを使います。

```text
コンテナA ─┐
コンテナB ─┼-> Redisの共有カウンター
コンテナC ─┘
```

どのコンテナへリクエストが届いても、同じクライアントのアクセス数を共有できます。

```json
{
  "RateLimit": {
    "Store": "Redis"
  }
}
```

Redisモードでは、カウンターの増加と有効期限設定をLuaスクリプトでまとめて実行しています。複数処理の途中で別のリクエストが割り込まないようにするためです。

## 8. レート制限の配置

このAPIでは、認証・認可の後にレート制限を実行します。

```text
Request ID
  ↓
アクセスログ
  ↓
認証
  ↓
認可
  ↓
レート制限
  ↓
エンドポイント
```

認証済みユーザーと未認証IPでパーティションキーを変えるため、認証後にユーザー情報が必要になります。

## 9. レート制限の注意点

レート制限は、APIを守る仕組みの一部です。

- 重要な書き込みAPIは厳しくする
- 読み取りAPIと書き込みAPIで制限を分けることを検討する
- 本番ではロードバランサーやWAFの制限も組み合わせる
- Redis障害時に無制限へフォールバックしない
- `Retry-After`をクライアントが扱えるようにする

## 練習問題

次の設定で、同じクライアントが何回目に拒否されるか考えてみてください。

```json
{
  "PermitLimit": 3,
  "WindowSeconds": 5
}
```

確認するポイント:

- 何秒間のカウンターか
- 何回まで許可されるか
- 拒否時のステータスコード
- 再試行時期をどのヘッダーで伝えるか
- 2コンテナ構成でMemoryモードを使った場合の問題
